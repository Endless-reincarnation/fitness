# 健身计划执行助手产品方案

## 1. 产品定位

本产品是一个面向健身新手和初中级训练者的微信小程序，核心目标是把健身计划变成可执行、可记录、可复盘的训练流程。

第一版不做课程内容平台，也不做社区型健身产品。产品重点放在训练执行闭环：

- 用户选择或创建计划。
- 系统拆成训练日和动作。
- 用户按动作完成训练。
- 系统提供动作拆解、肌群展示、休息计时和训练记录。
- 用户通过历史记录和体重趋势复盘进展。

## 2. 目标用户

第一阶段重点服务：

- 新手增肌用户。
- 刚开始去健身房但不知道如何执行计划的人。
- 收藏了很多训练计划但难以坚持的人。
- 想记录重量、次数、体重变化，但不想使用复杂 App 的用户。

暂不重点服务：

- 专业健美运动员。
- 竞技力量举用户。
- 重饮食管理用户。
- 直播课、跟练课用户。

## 3. 核心差异点

### 3.1 从看内容变成执行计划

多数健身产品偏课程、视频和内容消费。本产品偏训练执行：

- 今天练什么。
- 每个动作做几组。
- 每组做多少次。
- 组间休息多久。
- 当前重量是多少。
- 下次是否应该加重。
- 训练后身体数据是否有变化。

### 3.2 外部计划标准化

用户可以把看到的训练计划手动录入小程序。系统通过动作库把不同来源的计划统一映射成标准动作。

例如用户录入“哑铃上斜卧推”，系统映射到标准动作后展示：

- 主肌群。
- 辅助肌群。
- 器械。
- 动作步骤。
- 常见错误。
- 替代动作。

### 3.3 多计划库

官方计划不是文章，而是可执行模板：

- 计划目标。
- 适合人群。
- 难度。
- 周期。
- 每周训练次数。
- 训练日。
- 动作顺序。
- 组数。
- 次数范围。
- RPE。
- 休息时间。
- 替代动作。

### 3.4 训练记录驱动递进

第一版先做轻规则：

- 连续完成目标次数，提示下次小幅加重。
- 未完成目标次数，建议维持重量。
- RPE 过高，提示降低重量或延长休息。
- 训练中断，保留进度，下次继续。

## 4. 内容和版权策略

第一版不直接搬运未经授权的博主内容。

禁止直接使用：

- 博主视频。
- 博主口播文案。
- 博主封面。
- 博主头像。
- 博主昵称。
- 原始课程名称。
- 原始计划完整文案。

允许使用：

- 用户自己手动录入并私人使用的计划。
- 通用训练结构，例如三分化、推拉腿、上下肢、5x5。
- 自研动作库和自写动作说明。
- 自研、改写或授权的官方计划。
- 后续与博主合作发布授权计划。

长期方向可以发展为博主计划执行和分发工具，为博主提供专属计划码、计划执行页和训练反馈数据。

## 5. MVP 功能范围

### 5.1 小程序端

首页：

- 今日训练。
- 当前计划。
- 训练进度。
- 最近体重。
- 最近一次训练记录。

计划库：

- 官方计划列表。
- 按目标、难度、器械、每周训练次数筛选。
- 查看计划详情。
- 启用计划。

自定义计划：

- 新建计划。
- 新建训练日。
- 添加动作。
- 设置组数、次数、休息时间。
- 保存为自己的计划。
- 复制官方计划后修改。

训练执行：

- 当前训练日动作列表。
- 动作详情。
- 目标肌群。
- 每组重量、次数、RPE。
- 休息计时。
- 暂停和继续训练。
- 完成训练总结。

动作库：

- 动作搜索。
- 肌群筛选。
- 器械筛选。
- 动作步骤。
- 常见错误。
- 替代动作。

数据页：

- 训练历史。
- 体重记录。
- 体重趋势。
- 近 7 天和 30 天训练次数。
- 最近训练容量变化。

我的：

- 用户资料。
- 训练目标。
- 当前计划。
- 历史计划。
- 器械条件。

### 5.2 后台管理端

动作管理：

- 新增动作。
- 编辑动作。
- 设置肌群。
- 设置器械。
- 设置替代动作。
- 上架和下架动作。

计划管理：

- 新增官方计划。
- 编辑计划信息。
- 新增训练日。
- 编排动作。
- 设置组数、次数、RPE、休息时间。
- 上架和下架计划。
- 管理计划版本。

用户数据概览：

- 用户数。
- 启用计划数。
- 训练完成次数。
- 体重记录次数。
- 常用计划排行。

## 6. 数据模型

核心表：

- `users`：用户资料。
- `muscles`：肌群。
- `exercises`：动作库。
- `exercise_alternatives`：替代动作关系。
- `plan_templates`：官方计划模板。
- `plan_template_versions`：计划版本。
- `plan_days`：计划训练日。
- `plan_day_exercises`：训练日动作编排。
- `user_plans`：用户启用的计划实例。
- `custom_plans`：用户自定义计划。
- `workout_sessions`：一次训练。
- `workout_sets`：每组训练记录。
- `body_weights`：体重记录。
- `admin_logs`：后台操作日志。

关键原则：

- 官方计划是模板。
- 用户执行的是计划实例。
- 模板更新不影响历史记录。
- 动作只下架，不物理删除。
- 外部计划统一映射到自有动作库。
- 中文内容统一按 UTF-8 维护。

## 7. 第一批计划库建议

第一版建议先做 10 到 15 个官方计划：

- 新手三分化增肌。
- 新手全身训练。
- 推拉腿入门。
- 上下肢分化。
- 居家徒手入门。
- 哑铃居家增肌。
- 器械房入门。
- 减脂塑形基础。
- 女性臀腿塑形。
- 肩背改善。
- 下肢力量强化。
- 核心稳定训练。
- 久坐体态改善。
- 引体向上提升。
- 卧推基础提升。

每个计划都必须可直接执行，而不是只写说明。

## 8. 技术方案

第一版默认采用：

- 微信小程序。
- 微信云开发。
- 云数据库。
- 云函数。
- 云存储。
- 简单 Web 管理端。

选择该方案的原因：

- MVP 上线快。
- 微信登录天然支持 openid。
- 不需要一开始维护独立服务器。
- 适合先验证产品闭环。

后续用户规模上来后，再评估是否迁移到独立后端。

## 9. 视觉和皮肤

第一版默认视觉使用黑黄力量风：

- 黑色训练舱背景。
- 黄色作为主行动、重量、递进建议和关键数据颜色。
- 深色卡片和粗边框强化力量训练工具感。
- 页面重点突出重量、次数、RPE、休息计时和递进建议。

同时保留皮肤切换能力：

- `power-yellow`：黑黄力量风，默认皮肤。
- `tech-green`：黑绿科技风，作为备选皮肤。

皮肤切换第一版先存本地缓存，后续接云数据库后同步到 `users.theme`。

## 10. 验收标准

第一版完成时，需要满足：

- 用户能选择一个官方计划并启用。
- 用户能手动创建一个自定义计划。
- 用户能进入今日训练。
- 用户能查看动作拆解和肌群。
- 用户能记录每组重量、次数、RPE。
- 用户能使用休息计时器。
- 用户能中断并继续训练。
- 用户能完成训练并查看总结。
- 用户能记录体重并查看趋势。
- 后台能维护动作和计划。
- 官方计划更新不破坏历史训练记录。
- 用户能在“我的”页切换训练皮肤。

## 11. 产品风险

主要风险：

- 直接搬运博主内容存在版权风险。
- 计划质量差会影响用户信任。
- 新手动作执行错误可能带来安全风险。
- 如果计划录入太麻烦，用户流失会高。
- 如果没有递进反馈，长期留存弱。

对应策略：

- 官方内容坚持自研、改写或授权。
- 动作说明加入安全提示和常见错误。
- 第一版自定义计划录入流程尽量简单。
- 优先做好训练执行闭环。
- 后续加入文本解析和博主授权计划。
