# 算法规则说明

本文档统一记录项目中用于训练、饮食和 AI 计划生成的核心规则。后续调整算法时，应同步更新本文档，避免规则散落在页面和服务代码中。

## 1. 训练递进建议

代码位置：

- `miniprogram/utils/trainingInsight.js`

核心方法：

- `buildProgressionSuggestions(records)`
- `estimateOneRepMax(weightKg, reps)`
- `getSuggestedNextWeight(weightKg, role)`
- `roundTrainingWeight(weightKg)`

### 1.1 次数区间解析

从计划动作的 `targetReps` 中提取数字：

- `8-12` 解析为 `minTarget = 8`，`maxTarget = 12`
- `10` 解析为 `minTarget = 10`，`maxTarget = 10`
- 解析失败时兜底为 `1`

### 1.2 单组强度估算

使用 Epley 估算公式：

```text
估算 1RM = 重量 × (1 + 次数 / 30)
```

规则：

- 次数最多按 12 次计算，避免高次数组把估算值放大过多。
- 未记录重量或次数时，估算值为 0。
- 该值只用于比较同一动作不同组的相对强度，不作为严肃力量测试结果。

### 1.3 双进阶建议

当前采用“双进阶”思路：

1. 先在当前重量下把每组次数推进到目标区间上限。
2. 全部有效组达到上限后，再小幅加重。
3. 加重后如果首组低于目标下限，应回到上次重量。

建议类型：

- `track_weight`：没有外部重量记录。建议先稳定次数，若是可负重动作再记录重量。
- `increase`：所有有效组达到次数上限且完成度不低于 90%。建议小幅加重。
- `add_reps`：接近次数上限但未全组达成。建议维持重量，每组争取多 1 次。
- `reduce`：存在低于目标下限的组，或完成度低于 75%。建议降重约 5% 或延长休息。
- `hold`：默认维持重量，继续稳定完成目标次数。

### 1.4 加重和降重幅度

加重估算：

```text
主项：当前最佳重量 × 1.035
辅助/孤立项：当前最佳重量 × 1.025
```

降重估算：

```text
建议重量 = 当前最佳重量 × 0.95
```

重量统一按 `0.5kg` 取整。

## 2. 训练洞察

代码位置：

- `miniprogram/utils/trainingInsight.js`
- `miniprogram/pages/data/index.js`
- `miniprogram/pages/exercise-detail/index.js`

### 2.1 本周训练完成度

当前周按周一到周日计算。

目标天数来源优先级：

1. 当前启用计划的 `weeklyFrequency`
2. 当前启用计划的 `totalDays`
3. 当前计划 `days.length`
4. 兜底为 4

目标天数限制在 `1-7`。

完成度：

```text
完成度 = 本周有训练记录的天数 / 目标天数
```

同一天多次训练只算 1 个完成日，但组数和容量会累计。

### 2.2 动作进步榜

逻辑：

1. 按动作 ID 聚合历史训练记录。
2. 每次训练内取该动作“单组表现”最高的一组。
3. 对比该动作最近两次训练的最佳单组表现。
4. 按变化值排序展示前 5 个动作。

单组表现当前使用：

```text
单组表现 = 重量 × 次数
```

说明：

- 这是给用户快速感知趋势的轻量指标。
- 后续可升级为估算 1RM、总容量或多指标综合评分。

### 2.3 动作详情表现

动作详情页展示：

- 最近一次表现
- 训练次数
- 最高重量
- 最高次数
- 估算 1RM
- 最近一次训练生成的下次建议

估算 1RM 仅作为动作表现参考，不等同于真实最大力量测试。

## 3. 饮食与营养估算

代码位置：

- `miniprogram/services/aiPlanService.js`
- `miniprogram/services/planService.js`
- `miniprogram/utils/nutritionInsight.js`
- `miniprogram/pages/nutrition/index.js`

当前饮食规则是训练计划辅助说明，不是医学或营养处方。

### 3.1 热量估算

按体重和目标估算：

```text
增肌 / 增重：每日热量 = 体重 kg × 38
其他目标：每日热量 = 体重 kg × 28
```

### 3.2 蛋白质估算

```text
增肌 / 增重：蛋白质 g = 体重 kg × 2.0
其他目标：蛋白质 g = 体重 kg × 1.6
```

### 3.3 脂肪估算

```text
脂肪 g = 体重 kg × 0.9
```

### 3.4 碳水估算

```text
碳水 g = (每日热量 - 蛋白质 g × 4 - 脂肪 g × 9) / 4
```

结果四舍五入为整数。

### 3.5 体重兜底

AI 计划或官方计划动态营养估算缺失体重时，默认按 `70kg` 估算。有效体重范围一般限制在 `30-200kg`。

### 3.6 测试入口

营养估算规则通过以下命令回归：

```bash
npm run test:nutrition
```

## 4. 执行节奏与休息日

代码位置：

- `miniprogram/utils/executionInsight.js`
- `miniprogram/pages/home/index.js`

核心方法：

- `buildExecutionInsight(options)`
- `buildWeeklySchedule(plan, activePlan, history, baseDate)`
- `buildTodayAdvice(options)`
- `buildNutritionFocus(type)`

### 4.1 周节奏生成

首页不要求用户先手动排课，第一版按 `weeklyFrequency` 自动生成周一到周日节奏：

- 2 练：周一、周四训练，周六灵活
- 3 练：周一、周三、周五训练，周日灵活
- 4 练：周一、周二、周四、周五训练，周日灵活
- 5 练：周一、周二、周三、周五、周六训练
- 6 练：周一到周六训练，周日休息
- 7 练：每天训练

训练日名称来自当前计划 `days`，按顺序映射到周节奏训练位。

用户可在首页“本周安排”中临时调整每天类型：

- `training`：训练
- `rest`：休息
- `flex`：灵活

第一版调整结果存入本地缓存 `weeklyScheduleOverrides_${planId}`。覆盖默认节奏后，训练位会重新按周内顺序映射计划训练日；后续稳定后再同步到 `user_plans`。

### 4.2 今日建议

今日建议输出三类：

- `training`：建议训练
- `rest`：建议休息
- `optional`：可灵活安排或可补练

输出字段包含：

- `completedDays`：本周已完成训练天数
- `targetDays`：本周目标训练天数
- `remainingDays`：本周剩余目标次数
- `progressPercent`：首页本周目标进度条百分比

判断优先级：

1. 没有训练历史：建议从第 1 天开始训练。
2. 距离上次训练不足 20 小时：建议休息。
3. 本周完成天数已达到目标：建议休息。
4. 今天是训练位且恢复时间足够：建议训练。
5. 今天是休息位但本周落后且恢复时间足够：可补练。
6. 今天是灵活位：按本周完成进度判断训练或休息。

首页首屏根据 `todayAdvice.type` 切换标题：

- 训练日显示当前训练日名称。
- 休息日显示“今天建议休息”，副标题保留当前训练日，避免用户丢失下一次要练什么。
- 灵活日显示“今天可灵活安排”，并提示本周还差几次。

首页底部训练入口规则：

- 训练日直接进入训练页。
- 灵活日直接进入训练页，按钮文案为“灵活训练”。
- 休息日按钮文案为“仍要训练”，点击后先确认；确认后按当前训练日开始，完成后继续推进到下一训练日。

训练页会把入口类型写入草稿和训练记录：

- `entryAdviceType=training`：计划训练
- `entryAdviceType=optional`：灵活训练
- `entryAdviceType=rest`：休息日仍训练，同时 `isRestDayTraining=true`

### 4.3 首页营养重点

今日建议同步输出营养重点：

- 训练日：蛋白保持，碳水正常或略高，训练前后补充碳水和优质蛋白。
- 休息日：蛋白不降，碳水略收，蔬菜、饮水和睡眠优先。
- 灵活日：如果训练按训练日吃，如果休息则晚餐碳水可适当减少。

该逻辑目前只输出执行建议和文案，不直接改每日热量数值。

首页饮食摘要可进入 `pages/nutrition/index`。营养页第一版复用计划中的 `nutrition` 和 `nutritionInsight` 估算结果，展示每日热量、蛋白质、碳水、脂肪、今日饮食重点和食物换算；暂不记录实际摄入，后续再扩展每日饮食日志。

营养页支持训练日、休息日、灵活日切换：

- 训练日：使用完整计划目标。
- 休息日：蛋白和脂肪保持，碳水下调约 15%，总热量按宏量营养素重新估算。
- 灵活日：暂按训练日目标展示，用户如果选择休息可手动切到休息日。

## 5. BMI

代码位置：

- `miniprogram/pages/data/index.js`

公式：

```text
BMI = 体重 kg / (身高 m × 身高 m)
```

分类：

- `< 18.5`：体重偏轻
- `18.5 - 23.9`：正常体重
- `24 - 27.9`：体重超重
- `>= 28`：肥胖

说明：

- BMI 仅用于体重趋势辅助展示。
- 对高肌肉量用户可能存在误判。

## 6. AI 计划生成规则

代码位置：

- `miniprogram/services/aiPlanService.js`

### 6.1 模型配置

当前配置：

```text
AI_PROVIDER = hunyuan-v3
AI_MODEL = hy3-preview
```

调用方式：

- 优先使用小程序端 `wx.cloud.extend.AI`
- 优先尝试 `streamText`
- 不支持流式时回退 `generateText`
- 调用失败时回落本地 mock 草稿

### 6.2 输入约束

AI 表单收集：

- 计划名称
- 训练目标
- 训练水平
- 每周训练次数
- 计划周期
- 可用器械
- 限制和偏好
- 选填个人信息：性别、年龄、身高、体重

### 6.3 动作候选

AI 生成时使用标准动作库候选：

- 只使用结构化动作 ID 作为计划动作引用。
- 中文动作名称主要用于展示，不作为长期引用字段。
- 动作候选来自云端动作库，读取失败时回落本地 mock 动作。

### 6.4 输出结构

模型必须返回可直接进入自定义计划编辑页的 JSON：

- `name`
- `goal`
- `level`
- `durationWeeks`
- `weeklyFrequency`
- `equipmentTags`
- `summary`
- `overview`
- `generationSteps`
- `tips`
- `nutrition`
- `days`

每个训练日内动作必须包含：

- `exerciseId`
- `sets`
- `reps`
- `rpe`
- `restSeconds`
- `role`
- `roleLabel`
- `weightRule`
- `progressionRule`
- `notes`

### 6.5 AI 结果归一化

模型返回后会做结构校正：

- 训练日为空时抛错并回落本地 mock。
- 缺失营养字段时使用本文档第 3 节的饮食估算公式补齐。
- 缺失训练说明时补默认说明。
- 训练动作最终仍进入自定义计划编辑页，由用户确认后保存。

## 7. 后续算法管理要求

- 新增算法必须优先放在 `miniprogram/utils/` 或服务层公共方法中，避免直接散落在页面。
- 涉及训练、饮食、AI 生成策略变化时，必须同步更新本文档。
- 用户可见的建议需要说明“依据”和“下次目标”，避免只给黑盒结论。
- 高风险健康建议不在第一版提供，所有饮食和 BMI 内容仅作为训练辅助参考。

训练和营养算法统一使用以下命令回归：

```bash
npm run test:algorithms
```
